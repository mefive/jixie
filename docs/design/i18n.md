# 多语言(i18n)详设 —— 中英双语

> 2026-07-07 制定。目标:产品支持**中文 / 英文**。本文档是 i18n 工作的执行真相源;ROADMAP 里有指针。

## 两条硬需求(用户原话拆解)

1. **面向用户的中文字符串 → 走 i18n**(代码里不再硬编码中文,改由 key/变量取,值可 zh/en 切换)。
2. **所有给 LLM 的 prompt / system-prompt / 工具描述 / few-shot 一律英文**(和「代码全英文、维护者不识中文」一致);但 **LLM 的回复跟随用户提问的语言**(英文提问→英文回答)。
3. 附加:**代码注释一律英文**(维护者可能不识中文),用标准英文财经术语,不留中文括注。

### 需求 1 与需求 2 的边界(别搞混)

| 文本类别 | 语言策略 | 落点 |
| --- | --- | --- |
| UI 文案(按钮/标签/占位符/toast) | i18n,zh/en 切换 | 前端 `src/i18n` 资源 |
| 后端面向用户的报错 / 响应 / 回复外壳 | i18n,按请求 locale 渲染 | `apps/api/src/i18n` 消息目录 |
| **LLM system-prompt / 工具 description / few-shot 示例** | **一律英文,不 i18n** | 英文字面量,不走 i18n key |
| LLM 的**回复正文** | 跟随用户提问语言 | prompt 里一句中文指令控制,无需传 locale 给模型 |
| 代码注释 | **英文** | 就地改 |
| i18n 资源里的 zh 值、文档 md、commit message | 中文(正常) | —— |

> 红线:**绝不要把 prompt 文本抽成 i18n key**。prompt 是给模型的指令脚手架,用英文写(便于维护、代码全英文);A 股专有名词用标准英文财经术语。回复语言是另一根轴,靠 prompt 里一句英文指令「reply in the same language the user wrote in」实现,**不给模型传 locale**(模型从用户消息自己判断)。唯一按 locale 变的是「生成的名称语言」(命名助手按 Accept-Language 选中/英文名)。

## 架构

### locale 的流动

```
用户在顶栏切换 → localeStore(前端单例, 存 localStorage)
  ├─ 驱动 i18next(UI 文本)
  ├─ 驱动 antd ConfigProvider(组件内置文案)
  └─ api client 每个请求带 Accept-Language: zh|en
        └─ 后端 localeFromRequest(c) → t(locale, key) 渲染面向用户的字符串
        └─ agent turn 记下 locale,用于代码生成的回复外壳;
           模型回复正文的语言由 prompt 指令(跟随用户提问)决定,不依赖这个 locale
```

`Locale = 'zh' | 'en'` 定义在 `@jixie/shared`(`src/i18n.ts`),前后端共用;默认 `'zh'`。

### 前端(`apps/web/src/i18n/`)

- 方案:**react-i18next + antd ConfigProvider**(zhCN/enUS)。
- `index.ts` —— 初始化单个 i18next 实例;`lng` 取 localStorage(默认 zh)。
- `locale-store.ts` —— MobX 单例 `localeStore`:`locale` observable + `setLocale()`(持久化 + `i18n.changeLanguage` + `document.lang`)。**切换语言只调它**。
- `antd-locale.ts` —— `locale → zhCN|enUS`。
- `locales/<lng>/<namespace>.ts` —— 资源按**命名空间**分文件;**zh 是形状真相源**,en 用 `typeof zhX` 约束结构一致,漏 key 直接编译报错。
- 组件里用 `const { t } = useTranslation('<ns>')`;store 等非组件里用默认导出的 `i18n.t(...)`(store 字符串会在下次 action 重算,不追求即时重渲染)。

**命名空间划分**:`common`(共享 chrome)+ 一页一个(`lab` / `factor` / `screen` / `stock` / `login`)+ 领域(`agent`、`sdk` 文档、`learn` 教程)。

**key 命名**:语义分层小驼峰,`nav.backtest`、`error.saveFailed`、`trade.detailTitle`;带变量用 `{{count}}` 插值。

### 后端(`apps/api/src/i18n/`)

- `locale.ts` —— `localeFromRequest(c)` 读 Accept-Language,默认 zh。
- `messages.ts` —— `MESSAGES` 目录(英文 key → `{ zh, en }`)+ `t(locale, key, params?)`(`{name}` 占位)。**只放面向用户文本,不放 prompt**。
- 报错统一经 `lib/httpError.ts` 的 `apiError`;调用处传 `t(localeFromRequest(c), 'xxx')`。

### shared / DB 里的标签(key 化策略)

- `packages/shared` 的 `SCREEN_FIELDS`:label 迁到前端 i18n(按 field key 映射),shared 只留 `key` + 单位种类。
- `apps/api` 的 `builtin-factors.ts`:内置因子 label 是**中文规范名**,同时写进 DB 且作为 LLM 上下文——**DB 存的规范名不动**(它是数据+喂模型),前端**显示层**按 factor key 映射 en,取不到就回退 DB 名。避免 DB 迁移。

## 执行阶段(7 段)

- **Phase 0 基建**(本次):shared `Locale`;前端 i18n 骨架 + 顶栏切换 + Accept-Language;后端 `i18n/` + 验证器接入;本文档 + 约定。✅
- **Phase 1 prompt 层**:所有 prompt / system-prompt / 工具描述 / few-shot / model-facing 注入消息译成英文;回复语言改「跟随用户提问」(REPLY_LANGUAGE 英文指令);命名助手按 locale 选名称语言。
- **Phase 2 后端面向用户字符串**:`routes/*`、`llm/*`、engine 报错/响应 → 消息 key;`core.ts` 回复外壳按 turn locale。
- **Phase 3 前端外壳逐页**:components → login → lab → factor → screen → stock →(dashboard)+ 内置因子 en 映射 + SCREEN_FIELDS。
- **Phase 4 大段文档译英**:`learn.tsx` / `sdk-doc.tsx` / `sdk-reference.ts` 内容进 i18n 并人工校对级译英。
- **Phase 5 验证**:typecheck + build + eslint/prettier;Playwright e2e 跑 zh/en 双语并贴截图;实测英文 prompt→英文回复。
- **Phase 6 注释全英文**(贯穿):中文注释译英,用标准财经术语;跟各 Phase 文件改动合并进行。收尾 `grep` 复查残留中文只落在白名单(i18n zh 值 / prompt / md)。

## 新增一个命名空间的清单(给后续会话)

1. `locales/zh/<ns>.ts` 写 zh 形状;`locales/en/<ns>.ts` 用 `typeof` 约束写 en。
2. 两个 `locales/<lng>/index.ts` 各加一行注册。
3. 组件 `useTranslation('<ns>')`,逐个中文字面量换 `t('key')`;原中文进 zh 值。
4. 该文件的中文注释顺手译英(Phase 6 合并)。
